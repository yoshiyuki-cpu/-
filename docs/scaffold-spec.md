# 足場計算システム 設計書

- 対象機能: 足場計算（現状パス `/scaffold`。本設計ではこれを維持しつつ、現場に紐付いた保存版 `/projects/[id]/scaffold` を新設する）
- 対象ファイル: [app/scaffold/page.tsx](../app/scaffold/page.tsx)（現状実装）
- 作成日: 2026-07-23
- ステータス: Phase 1 実装・動作確認完了（2026-07-23）。次はPhase 2（図面トレース入力）の着手判断待ち

## 1. 背景・目的

現在の足場計算機能は、建物を「東西南北4辺」または「縦×横の長方形」「周囲長のみ」の3パターンで入力し、辺の長さ÷スパン間隔・高さ÷段の高さを切り上げてスパン数・段数・単管本数を概算する電卓ツール。

実際の解体現場は長方形とは限らず、増築部・凹凸・離れなどがある建物が多い。現状の入力方法では実際の建物形状を近似できず、本数の見積り誤差が大きくなる。すでに現場マスタには上空図面（`projects.aerial_photo_url`）をアップロードする機能があるため、この図面（または現場で別途撮影・アップロードする図面写真）の上で建物の輪郭をなぞって入力できるようにし、より正確な足場材料の拾い出しを可能にする。

**Why:** 見積り・資材発注の精度を上げ、単管の過不足（現場での急な追加発注・余剰在庫）を減らしたい。
**How to apply:** この機能に着手する際はこのドキュメントを起点に、Phase 1から順に実装する。

## 2. 現状の整理

### 2.1 現状の入力モード（3種類、`InputMode`）

| モード | 入力内容 | 備考 |
|---|---|---|
| `directions`（東西南北） | 東西南北それぞれの長さ・高さ | 辺ごとに高さが違う前提で計算できる唯一のモード |
| `rect`（縦×横） | 縦・横のみ | 高さは建物全体で1つ、周囲長 = (縦+横)×2 |
| `perimeter`（周囲長） | 周囲長のみ | 高さは建物全体で1つ |

### 2.2 現状の計算ロジック

- 辺ごとに `spanCount = ceil(長さ / スパン間隔)`、`levelCount = ceil(高さ / 段の高さ)`
- `建地(tateji) = spanCount`（辺の始点ごとに1本、周を一周する想定）
- `布(nuno) = spanCount × levelCount`（一側足場）
- 単管の長さ別本数内訳は `pipeBreakdown()` で規格長（4/3/2/1m）から算出し、建地用・布用をそれぞれ集計
- 用途別本数（筋交い・手すり・幅木・ジョイント・ベース金具）は手入力のみで自動計算対象外

### 2.3 現状の制約・課題

1. **建物形状が長方形前提**：L字・凹凸・離れのある敷地は「東西南北」でも実際の形状を正しく表現できず、ユーザーが頭の中で辺を合成して概算入力する必要がある
2. **現場（project）に紐付いていない**：`/scaffold` はグローバルな電卓ページで、計算結果は保存されず、現場詳細画面や上空図面と連携していない
3. **出隅・入隅を区別していない**：凹凸のある形状では出隅（外側の角）と入隅（内側の角）で足場の掛け方・本数の考え方が変わるが、現状のロジックは考慮していない
4. **図面・写真を計算に使えない**：`projects.aerial_photo_url` は表示のみで、寸法入力の補助には使われていない

## 3. 新機能: 図面トレース入力

### 3.1 概要

既存の3モードに加えて、第4の入力モード「図面トレース」を追加する。

1. 現場の上空図面（または新規アップロードした敷地図・平面図の写真）を画面に表示
2. ユーザーが建物の角を順にタップして輪郭（多角形）をなぞる
3. 最初の2点を打った時点で「その区間の実寸（m）」を1回だけ入力してもらい、画像上のピクセル距離との比率からスケール（m/px）を算出する
4. 以降の頂点はスケールを使って自動的に辺の長さ（m）に変換される
5. 各辺（セグメント）ごとに高さ（m）を入力する（東西南北モードと同様、辺ごとに高さを変えられる）
6. 確定した多角形の各辺に対して、既存の `sideResults` と同じロジック（スパン数・段数・建地・布・単管長さ別本数）を適用する

これにより「東西南北」モードの上位互換として、任意の頂点数（矩形・L字・凹凸形状）に対応する。

### 3.2 スケール校正の仕組み

- 校正用の実寸入力は「外壁の1辺（例: 玄関側の幅）」など、現場でメジャーで測りやすい・図面に記載がある辺を想定
- 校正はプラン単位で1回のみ。校正済みの縮尺（m/px）を保存し、以降の頂点はその縮尺で自動換算
- 校正のやり直し（縮尺リセット）はUI上で可能にする（測り間違いに気づいた場合など）

### 3.3 出隅・入隅の扱い（Phase 2以降）

- Phase 1では現状踏襲でシンプルに「頂点1つ = 建地1本」として計算する（出隅・入隅を区別しない）
- Phase 2で、多角形の頂点の内角から出隅（内角 < 180°）・入隅（内角 > 180°）を判定し、入隅では建地を共有しない・増し本数にする等の補正を検討する（現場の実態に合わせて後日ヒアリングして係数を決める）

### 3.4 操作性（モバイル前提）

- 画像はピンチズーム・パン操作に対応する
- 頂点はタップで追加、直前の頂点まで線を引いて表示する
- 最後の頂点をタップし直すことで多角形を閉じる（始点近くをタップで自動スナップ）
- 打った頂点はドラッグで位置修正可能、末尾からのUndo（一つ前に戻す）ボタンを用意する
- 誤操作対策として「やり直し」「クリア」ボタンを用意する

## 4. 対象を現場（project）に紐付ける

現状 `/scaffold` は現場非依存のグローバル電卓のため、図面トレースを機能させるには現場の上空図面と連携する必要がある。あわせて計算結果の保存も可能にする。

### 4.1 ルーティング変更

- 既存の `/scaffold`（現場非依存・保存なしのグローバル電卓）はそのまま維持する。下部固定タブバーの「足場計算」タブも変更しない
- 新たに `/projects/[id]/scaffold` を追加し、現場詳細画面（[app/projects/[id]/page.tsx](../app/projects/[id]/page.tsx)）から遷移する導線（リンクボタン）を設置する。計算ロジック・UIは`/scaffold`とほぼ共通だが、現場の上空図面と連携し、入力内容を保存・再読込できる
- 1現場につき足場計算プランは1つのみとする（複数棟の現場は当面、東西南北モード等で概算合算するか、Phase 3以降で対応を検討）

### 4.2 図面ソース

- 初期表示は `projects.aerial_photo_url` があればそれを使う
- 上空図面が無い、または別の図面（平面図・立面図）を使いたい場合は、足場計算画面から専用に画像をアップロードできるようにする（`project-files` バケットの `scaffold/{project_id}/{plan_id}.{ext}` 等に保存。既存の画像縮小ヘルパー（[app/projects/[id]/minutes/page.tsx:12](../app/projects/[id]/minutes/page.tsx)の`resizeImageToBase64`と同様の考え方）を流用してアップロード前に縮小する）

## 5. データモデル

新規テーブルを追加する（マイグレーションSQLはリポジトリ直下に `supabase-schema-scaffold.sql` として保存し、Supabase SQL Editorでユーザーに実行してもらう運用は他機能と同様）。

### 5.1 `scaffold_plans`（足場計算プラン）

| カラム | 型 | 備考 |
|---|---|---|
| id | uuid, PK | |
| project_id | uuid, FK → projects.id, UNIQUE | 1現場1プランのためUNIQUE制約 |
| input_mode | text | `directions` \| `rect` \| `perimeter` \| `trace` |
| image_url | text, nullable | トレースに使った図面画像（未指定なら`projects.aerial_photo_url`を参照） |
| scale_m_per_px | numeric, nullable | 校正済み縮尺 |
| span_interval_m | numeric | スパン間隔（デフォルト1.8） |
| level_height_m | numeric | 段の高さ（デフォルト1.8） |
| created_at | timestamptz | |
| updated_at | timestamptz | |

### 5.2 `scaffold_segments`（辺・セグメント）

| カラム | 型 | 備考 |
|---|---|---|
| id | uuid, PK | |
| plan_id | uuid, FK → scaffold_plans.id | |
| order_index | int | 頂点順（多角形の周回順） |
| label | text, nullable | `directions`モード由来の場合は「東」等、traceモードはnull可 |
| length_m | numeric | 辺の長さ |
| height_m | numeric | この辺の足場の高さ |
| vertex_x_px | numeric, nullable | traceモードのみ：画像上の頂点座標（再編集用） |
| vertex_y_px | numeric, nullable | traceモードのみ |

計算結果（スパン数・建地・布・単管長さ別本数）は保存時に再計算可能なため永続化せず、都度算出する方針とする（既存の実行予算等と同様に「値の一次ソースは入力、集計は都度計算」の方針に揃える）。

## 6. 計算ロジックの変更点

- `sides: Side[]` の生成を、`directions`/`rect`/`perimeter` モードの分岐に加えて `trace` モード（`scaffold_segments`をそのまま`Side[]`に変換）を追加する形にする
- `sideResults` 以降（スパン数・段数・建地・布・単管本数内訳）のロジックは頂点数に依存しない設計になっているため、そのまま流用できる見込み（[app/scaffold/page.tsx:93](../app/scaffold/page.tsx)の`sideResults`計算部分）
- 多角形が3辺未満（=閉じていない）場合は計算対象外とし、「頂点が不足しています」等のガイダンスを表示する

## 7. 画面構成（案）

```
/projects/[id]/scaffold（1現場1プラン。初回アクセス時に自動でプランを作成/読込）
├─ 入力方法選択（東西南北 / 縦×横 / 周囲長 / 図面トレース）※現行3種 + 新規1種
├─ 図面トレース選択時:
│   ├─ 図面画像表示（上空図面 or アップロード）
│   ├─ トレースキャンバス（タップで頂点追加・ドラッグ修正・Undo・クリア）
│   ├─ スケール校正ダイアログ（2点目タップ時に1回表示）
│   └─ 頂点確定後、辺ごとの高さ入力リスト
├─ スパン間隔・段の高さ入力（既存のまま）
├─ 計算結果（辺別テーブル + 合計。既存のまま）
├─ 単管の長さ別本数（既存のまま）
├─ 用途別本数（手入力、既存のまま）
└─ 保存ボタン（scaffold_plans/scaffold_segmentsへ保存）
```

## 8. 非機能要件

- モバイル（スマホ）操作を前提とし、タップ精度が低くても頂点修正しやすいUI（ドラッグで微調整、ピンチズームで拡大してから頂点を打てるようにする）にする
- 図面画像はアップロード前に縮小（既存パターン踏襲、長辺1600px程度・JPEG品質0.7目安）してモバイルブラウザのメモリクラッシュを防ぐ
- オフライン対応は現状他機能同様に考慮しない

## 9. フェーズ分け

- **Phase 1**: `/projects/[id]/scaffold` を新設（既存の`/scaffold`は維持）し、`scaffold_plans`/`scaffold_segments`を新設して既存3モード（東西南北・縦横・周囲長）の入力内容を現場に保存・再読込できるようにする（トレース機能なし）
- **Phase 2**: 図面トレース入力（スケール校正・多角形入力・辺ごと高さ入力）を追加する
- **Phase 3**: 出隅・入隅の補正計算、資材コスト計算（単価×本数）など、Phase 1/2の運用後にニーズを見て検討する
